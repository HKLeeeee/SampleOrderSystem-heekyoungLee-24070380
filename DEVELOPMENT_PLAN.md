## 1. 전체 로드맵

```
Phase 0  PoC 4종 (개별 repo)          ── 검증: 구조/영속성/도구 동작
Phase 1  프로젝트 골격 + Harness       ── 검증: 빌드·테스트 1커맨드 실행
Phase 2  도메인 모델 + 상태머신        ── 검증: 단위 테스트 (전이/공식)
Phase 3  영속성 계층                   ── 검증: 재시작 후 데이터 유지
Phase 4  기능 구현 (FR-1 → FR-6)       ── 검증: 기능별 AC + 단위 테스트
Phase 5  통합·시나리오 검증            ── 검증: E2E 시나리오 (더미 데이터)
Phase 6  리팩토링·문서·최종 제출       ── 검증: 리뷰 체크리스트 전 항목
```

의존 관계: Phase 2 ← 3 ← 4, FR-3은 FR-1·FR-2 이후, FR-5·FR-6은 FR-3 이후.

## 6. Phase 4 — 기능 구현 (PRD FR 순서)

각 기능: ①테스트 작성 → ②Service 구현 → ③View/Controller 연결 → ④AC 수동 확인 → ⑤커밋

| 순서 | 기능           | 주요 구현                                                       | 검증 포인트 (PRD AC)                        |
| ---- | -------------- | --------------------------------------------------------------- | ------------------------------------------- |
| 4-1  | FR-1 시료 관리 | 등록/조회(+재고, 페이지)/검색                                   | 중복 ID 거부, 부분 일치 검색                |
| 4-2  | FR-2 시료 주문 | 입력 검증→확인→RESERVED 접수                                    | 미등록 시료 거부, 주문번호 형식             |
| 4-3  | FR-3 승인/거절 | 재고 자동 분기, 큐 등록, REJECTED                               | 충분→CONFIRMED, 부족→PRODUCING+큐, 206 계산 |
| 4-4  | FR-5 생산라인  | 진행 모델 구현(택1 문서화), 현황+대기열, 완료 시 재고+CONFIRMED | FIFO 순서, 완료 전환                        |
| 4-5  | FR-4 모니터링  | 상태별 집계(REJECTED 제외), 여유/부족/고갈                      | 재고0=고갈, REJECTED 미집계                 |
| 4-6  | FR-6 출고      | CONFIRMED 목록→재고 차감→RELEASE                                | 차감 정확성, 경합 차단                      |
| 4-7  | FR-0 완성      | 메인 요약 실데이터 연결                                         | 4종 요약 정합성                             |

※ FR-5를 FR-4보다 먼저 구현 — 모니터링이 PRODUCING/CONFIRMED 데이터를 필요로 함.

**게이트:** FR별 AC 전부 충족 + 단위 테스트 green

## 7. Phase 5 — 통합·시나리오 검증

DummyDataGenerator(PoC4)로 시료 12종/주문 36건 시딩 후 E2E 시나리오 수행:

```
S1 정상 흐름(재고 충분): 등록→주문→승인(CONFIRMED)→출고(RELEASE)→재고 차감 확인
S2 생산 흐름(재고 부족): 주문→승인(PRODUCING,큐)→생산완료(CONFIRMED,재고↑)→출고
S3 거절 흐름: 주문→거절(REJECTED)→모니터링 미집계 확인
S4 FIFO: 부족 주문 3건 연속 승인→큐 순서·완료 순서 일치
S5 영속성: S1~S4 후 재시작→전체 상태·큐 유지
S6 예외: 잘못된 입력 전 메뉴 투어→비정상 종료 없음
```

DataMonitor(PoC3)로 외부에서 데이터 상태 교차 검증.
**게이트:** S1~S6 전부 통과 (결과를 docs/test-scenarios.md 에 기록)

## 9. 테스트 전략 요약

| 레벨   | 대상                                                                                                                                   | 도구/방법                         |
| ------ | -------------------------------------------------------------------------------------------------------------------------------------- | --------------------------------- |
| 단위   | 상태 전이, 생산 계산, FIFO, 주문번호, 재고 차감                                                                                        | 테스트 프레임워크 (harness 포함)  |
| 통합   | Service+Repository (영속성 경유)                                                                                                       | 임시 데이터 파일 사용             |
| E2E    | S1~S6 시나리오                                                                                                                         | 더미 시딩 + 수동/스크립트         |
| 회귀   | 매 커밋 전 전체 harness                                                                                                                | `make test` 등 1커맨드            |
| 안정성 | 신규 기능의 견고성 — 비정상 입력, 경계값, 금지된 상태 전이, 데이터 파일 손상에도 시스템이 비정상 종료·데이터 오염 없이 동작하는지 검증 | 의도적 파괴 시도(네거티브 테스트) |

### 안정성 검증 체크리스트 (신규 기능마다 수행)

- [ ] 입력 공격: 숫자 자리에 문자/공백/음수/0/극대값 입력 → 오류 안내 후 재입력 (크래시 금지)
- [ ] 상태 공격: 금지된 전이 시도(예: RELEASE 주문 재출고, REJECTED 주문 승인) → 예외 처리 확인
- [ ] 데이터 공격: 영속 파일 삭제/내용 손상 후 재실행 → 정의된 복구 정책대로 동작
- [ ] 경계값: 재고 == 주문량, 부족분 1, 수율 하한·상한 등 분기 경계 직접 타격
- [ ] 반복 공격: 동일 주문 중복 승인/중복 출고 시도 → 재고·상태 정합성 유지

## 10. 커밋 계획 (Conventional Commits)

- Phase/작업 번호 단위로 커밋: `feat: 주문 승인 재고 분기 구현 (FR-3.2)`
- 테스트는 `test:`, 골격/도구는 `build:`/`chore:`, 문서는 `docs:`
- 1커밋 = 1관심사, harness green 상태에서만 커밋
