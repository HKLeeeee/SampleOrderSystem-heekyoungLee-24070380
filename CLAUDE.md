## 1. 프로젝트 개요

가상의 반도체 회사 **S-Semi** 의 시료(Sample) 생산·주문·재고·출고를 관리하는
**콘솔 기반** 시스템. 담당자가 직접 명령(메뉴 번호)을 입력하여 동작한다.

**역할 흐름:** 고객(이메일로 시료 요청) → 주문 담당자(주문서 작성/접수) → 생산 담당자(승인·거절, 생산, 출고)

---

## 2. 도메인 모델

### 2.1 시료 (Sample)
| 속성 | 설명 |
|---|---|
| 시료 ID | 고유 식별자 (예: `S-001`) |
| 이름 | 예: `SiC 파워기판-6인치` |
| 평균 생산시간 | 단위: min/ea (예: 0.8) |
| 수율(yield) | 정상 시료 / 총 생산 시료 (예: 0.92) |
| 현재 재고 | ea 단위 |

- **시스템에 등록된 시료만 주문 가능**

### 2.2 주문 (Order)
- 입력 값: **시료 ID, 고객명, 주문 수량**
- 주문번호 형식 예: `ORD-YYYYMMDD-NNNN`

### 2.3 주문 상태 머신 (절대 위반 금지)
```
RESERVED ──거절──> REJECTED (종결, 모니터링 제외)
RESERVED ──승인──┬─ 재고 충분 ──> CONFIRMED ──출고──> RELEASE
                 └─ 재고 부족 ──> PRODUCING ──생산완료──> CONFIRMED ──출고──> RELEASE
```
| 상태 | 의미 |
|---|---|
| RESERVED | 주문 접수 |
| REJECTED | 주문 거절 (정상 흐름 외, 모니터링 제외) |
| PRODUCING | 승인 완료 + 재고 부족으로 생산 중 |
| CONFIRMED | 승인 완료 + 출고 대기 |
| RELEASE | 출고 완료 |

### 2.4 생산 라인 (Production Line)
- 공장에 **단일 라인**, 시료를 **하나씩(한 주문씩)** 생산
- **주문이 들어온 시료에 대해서만** 생산 (선제 생산 금지)
- 대기열: **FIFO 스케줄링**
- 핵심 공식 (변경 금지):
  - `부족분 = 주문량 - 현재 재고`
  - `실생산량 = ceil(부족분 / (수율 × 0.9))`
  - `총 생산시간 = 평균 생산시간 × 실생산량`
- 생산 완료 시: 주문 상태 `PRODUCING → CONFIRMED`, 재고 반영

## 3. 기능 명세 (메인 메뉴 6종)

메인 메뉴 진입 시 요약 정보 표기: 등록 시료 수, 총 재고, 전체 주문 수, 생산라인 대기 건수

### [1] 시료 관리
- 시료 등록: ID, 이름, 평균 생산시간, 수율 입력
- 시료 조회: 전체 목록 + 현재 재고 함께 표시 (페이지네이션 권장)
- 시료 검색: 이름 등 속성으로 검색

### [2] 시료 주문 (접수)
- 시료 ID / 고객명 / 주문 수량 입력 → 입력 내용 확인 → 예약 접수
- 접수 시 상태 = `RESERVED`, 주문번호 발급
- 재고 확인·승인은 이 단계에서 하지 않음 (승인 메뉴에서 처리)

### [3] 주문 승인/거절
- `RESERVED` 상태 주문 목록 표시 → 특정 주문 선택
- **승인 시 재고에 따라 자동 분기:**
  - 재고 충분 → 즉시 `CONFIRMED`
  - 재고 부족 → 생산 라인(큐)에 자동 등록 + `PRODUCING`
    - 부족분/실생산량/예상 생산시간을 사용자에게 표시 후 확정
- 거절 시 → 즉시 `REJECTED`

### [4] 모니터링
- 주문량 확인: 상태별(RESERVED / CONFIRMED / PRODUCING / RELEASE) 건수 — **REJECTED 제외**
- 재고량 확인: 시료별 재고 + 주문대비 상태 표기
  - `여유`: 주문대비 재고 충분 / `부족`: 주문대비 재고 부족 / `고갈`: 재고 0

### [5] 생산라인 조회
- 현재 생산 중인 주문 정보 표시 (주문번호, 시료, 주문량, 부족분, 실생산량, 진행률 등 — 표기 수준 자율)
- 대기 큐(FIFO 순) 목록: 주문번호, 시료, 주문량, 부족분, 실생산량, 예상 완료 시각

### [6] 출고 처리
- `CONFIRMED` 주문 목록 표시 → 선택 주문 출고 실행
- 재고 차감 + 상태 `CONFIRMED → RELEASE`, 처리일시 기록

## 4. 아키텍처 규칙

- **MVC 패턴 강제**: `model` / `view` / `controller` 패키지 분리
  - Model: 도메인 엔티티 + 비즈니스 로직 + 저장소(repository)
  - View: 콘솔 입출력 전담 (비즈니스 로직 금지)
  - Controller: 흐름 제어, Model↔View 중개
- **데이터 영속성**: 파일/JSON/DB 중 선택한 방식으로 재실행 후에도 데이터 유지 (CRUD 완비)
- 상태 전이는 반드시 단일 지점(예: `Order.changeState()` 또는 서비스 계층)에서만 수행 — 임의 필드 대입 금지
- 수율 공식, 상태 머신은 테스트로 고정(회귀 방지)

## 5. 코드 품질 규칙 (미션2 주안점)

1. **문서 관리**: `CLAUDE.md`(=본 AGENTS.md 참조), `PRD.md` 최신 상태 유지. 명세 변경 시 문서 먼저 수정
2. **Harness**: 빌드/실행/테스트를 한 번에 돌리는 스크립트(예: `make test`, `./gradlew test`, `run.sh`) 구비. 에이전트는 작업 후 반드시 harness 실행으로 검증
3. **Test**: 핵심 도메인 로직(상태 전이, 실생산량 계산, FIFO 큐, 재고 차감) 단위 테스트 필수. 테스트 없는 기능 추가 금지
4. **Clean Code**: 의미 있는 네이밍, 함수 단일 책임, 매직넘버 상수화(`0.9` 보정계수 등), 중복 제거
5. **Commit 이력**: 기능 단위의 작은 커밋. 메시지 규칙 `feat:`, `fix:`, `test:`, `refactor:`, `docs:` (Conventional Commits)

## 6. 작업 진행 단계 (권장 순서)

```
Phase 0. PoC 4종 (개별 repo) ─ §8 참조
Phase 1. 프로젝트 골격: MVC 패키지 + harness + CI 구성 (PoC 1 재활용)
Phase 2. 도메인 모델: Sample, Order, 상태머신 + 단위 테스트
Phase 3. 영속성 계층: Repository 인터페이스 + 구현 (PoC 2 재활용)
Phase 4. 기능 구현 (테스트 선행):
   4-1 시료 관리 → 4-2 주문 접수 → 4-3 승인/거절(+생산큐 등록)
   → 4-4 생산라인(FIFO, 생산완료 처리) → 4-5 모니터링 → 4-6 출고
Phase 5. 메인 메뉴/요약 화면 통합, Dummy 데이터로 시나리오 검증 (PoC 4 재활용)
Phase 6. 리팩토링 + 문서 정리 + 최종 커밋
```

## 7. 에이전트 작업 수칙

- 한 번에 하나의 Phase/기능만 작업하고, 완료 시 harness(테스트) 통과를 확인한 후 커밋
- 명세에 없는 기능을 임의로 추가하지 말 것 (UI 구성은 자율이나 동작 규칙은 명세 고정)
- 모호한 부분은 본 문서 §2~§3 을 우선 적용, 그래도 모호하면 사용자에게 질문
- 서브에이전트: `.claude/agents/` 의 PoC별 에이전트, `test-engineer`, `code-reviewer` 활용
- 도메인 규칙 상세는 skill `semiconductor-domain-rules` 참조
